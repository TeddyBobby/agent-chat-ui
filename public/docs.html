一、总览

PiAgent 是嵌入在 agent-chat-ui（Next.js 16）中的最小化 AI Coding Agent 框架。约 600 行 TypeScript，零外部 Agent 依赖，只靠 fetch 调 LLM API + Node.js 标准库。

二、类型系统

框架只暴露三个对外类型，其余都是内部实现细节。

Tool — 工具接口

interface Tool {
  name: string;           // 工具名，LLM 通过这个名字调用
  description: string;    // 告诉 LLM 这个工具干什么
  schema: {               // JSON Schema，LLM 据此生成参数
    type: "object";
    properties: Record<string, { type: string; description: string }>;
    required: string[];
  };
  run(args: Record<string, unknown>): Promise<string>;
}

AgentConfig — 配置

interface AgentConfig {
  apiKey: string;         // LLM API 密钥
  model: string;          // 模型名，如 gpt-4o / gemma4:e4b
  baseURL?: string;       // API 地址，默认 OpenAI
  maxSteps?: number;      // 最大推理步数，默认 15
  systemPrompt?: string;  // 自定义 system prompt
}

StepEvent — 流式回调事件

interface StepEvent {
  type: "thought" | "action" | "observation" | "answer";
  content: string;
  toolName?: string;                        // action/observation 时附带
  toolArgs?: Record<string, unknown>;      // action 时附带
}

Agent 内部还有 Message 类型（对齐 OpenAI API 的 role/content/tool_calls/tool_call_id 字段），不对外暴露。

三、Tool 接口 — 框架的扩展点

任何实现 Tool 接口的对象都可以通过 agent.use(tool) 注册。框架不关心工具内部做什么——读文件、调 API、操作数据库都行。

// 最小工具示例：获取当前时间
const timeTool: Tool = {
  name: "get_time",
  description: "获取当前时间",
  schema: {
    type: "object",
    properties: {},
    required: [],
  },
  async run(args) {
    return new Date().toISOString();
  },
};

agent.use(timeTool);

工具注册是链式调用的：agent.use(t1).use(t2).use(t3)，内部用 Map<string, Tool> 存储。

四、System Prompt

buildPrompt(tools, custom?) 构造发送给 LLM 的第一条消息：

你是一个 coding agent。你可以使用工具来读写文件、搜索代码、执行命令。

## 可用工具
- read_file: 读取文件内容，支持行号范围
- write_file: 写入文件（覆盖），自动创建父目录
- edit_file: 精准编辑文件——查找并替换指定文本
- search_code: 搜索代码库中的文本或正则
- run_command: 执行 shell 命令

## 工作方式
- HTML/CSS/JS 等网页文件请放在 public/ 目录下
- 使用 edit_file 精准编辑已有文件，避免全量重写
- 完成任务后简要总结你做了什么

如果传入 systemPrompt 配置项，跳过内置 prompt 直接用自定义的。

五、主循环 — run()

这是 Agent 的心脏，约 90 行。核心逻辑：

run(task, history?) │ ├─ 1. 构建 messages 数组 │ [system prompt] + [历史] + [task] │ ├─ 2. 转换工具 → OpenAI function calling 格式 │ Tool[] → [{ type: "function", function: {…} }] │ └─ 3. while step < maxSteps: │ ├─ callLLM(messages, tools) → /chat/completions │ ├─ 响应有 tool_calls？ │ ├─ YES → 遍历每个 tool_call │ │ ├─ 查找工具 → JSON.parse 参数 │ │ ├─ emit("action") → 通知外部 │ │ ├─ tool.run(args) → 执行 │ │ ├─ emit("observation") → 通知外部 │ │ └─ 结果推入 messages (role: "tool") │ └─ continue │ └─ 无 tool_calls → return content 作为答案

完整源代码（带注释）

async run(
  task: string,
  history?: Array<{ role: "user" | "assistant"; content: string }>
): Promise<string> {
  // 获取所有已注册的工具
  const toolList = Array.from(this.tools.values());
  
  // 构建初始消息：system prompt + 可选历史 + 用户任务
  const messages: Message[] = [
    { role: "system", content: buildPrompt(toolList, this.config.systemPrompt) },
  ];

  if (history && history.length > 0) {
    // 把历史对话文本化，注入到用户消息里
    const contextMsg = history
      .map(m => `[${m.role === "user" ? "用户" : "AI"}]: ${m.content}`)
      .join("\n\n");
    messages.push({
      role: "user",
      content: `以下是之前的对话历史：\n\n${contextMsg}\n\n---\n当前任务：${task}`,
    });
  } else {
    messages.push({ role: "user", content: task });
  }

  // 把 Tool 对象转换为 OpenAI function calling 所需的 tools 参数格式
  const openaiTools = toolList.map(t => ({
    type: "function" as const,
    function: {
      name: t.name,
      description: t.description,
      parameters: t.schema,   // JSON Schema 直接透传
    },
  }));

  // ---- ReAct 主循环 ----
  for (let step = 0; step < this.config.maxSteps; step++) {
    // ① 调用 LLM
    const data = await this.callLLM(messages, openaiTools);
    const msg: Message = data.choices[0].message;

    // ② LLM 要调工具 → 执行
    if (msg.tool_calls && msg.tool_calls.length > 0) {
      messages.push(msg);  // 把 LLM 的 tool_calls 消息记入历史

      for (const tc of msg.tool_calls) {
        const tool = this.tools.get(tc.function.name);
        if (!tool) {
          // 工具不存在 → 把错误作为 observation 返回给 LLM
          messages.push({ role: "tool", tool_call_id: tc.id, content: `工具不存在` });
          continue;
        }

        // 解析 LLM 生成的参数 JSON
        let args: Record<string, unknown> = {};
        try { args = JSON.parse(tc.function.arguments); }
        catch { /* JSON 解析失败 → 错误喂回 LLM */ }

        // 通知外部（route.ts 将其转为 SSE 事件）
        this.emit({ type: "action", content: `调用 ${tc.function.name}`,
                     toolName: tc.function.name, toolArgs: args });

        // ③ 执行工具
        try {
          const result = await tool.run(args);
          // 把结果作为 tool role 消息推入，LLM 下一轮能看到
          messages.push({ role: "tool", tool_call_id: tc.id, content: result });
          this.emit({ type: "observation", content: result, toolName: tc.function.name });
        } catch (e) {
          // 工具执行异常 → 错误喂回 LLM
          messages.push({ role: "tool", tool_call_id: tc.id, content: `异常: ${e.message}` });
        }
      }
      continue;  // 进入下一轮，LLM 看到工具结果后决定下一步
    }

    // ④ 没有 tool_calls → LLM 认为任务完成，返回最终答案
    const answer = msg.content || "";
    if (answer) {
      this.emit({ type: "thought", content: "任务完成" });
      this.emit({ type: "answer", content: answer });
    }
    return answer;
  }

  // 用尽步数 → 返回错误
  return `未完成：已达最大步数 ${this.config.maxSteps}`;
}

六、LLM 调用 — callLLM()

private async callLLM(
  messages: Message[],
  tools: Array<{
    type: "function";
    function: { name: string; description: string; parameters: ToolSchema };
  }>
): Promise<any> {
  const body: any = {
    model: this.config.model,
    messages,
    temperature: 0,
  };
  if (tools.length > 0) {
    body.tools = tools;
    body.tool_choice = "auto";  // LLM 自己判断是否需要调工具
  }

  const res = await fetch(`${this.config.baseURL}/chat/completions`, {
    method: "POST",
    headers: {
      "Content-Type": "application/json",
      Authorization: `Bearer ${this.config.apiKey}`,
    },
    body: JSON.stringify(body),
  });

  if (!res.ok) {
    const text = await res.text();
    throw new Error(`LLM API ${res.status}: ${text.slice(0, 300)}`);
  }

  return res.json();
}

temperature: 0 保证确定性——同一问题不会随机调不同工具。tool_choice: "auto" 让 LLM 自己判断是否需要调工具。

七、多轮对话

run() 接受可选的 history 参数。有历史时，消息格式：

[用户]: 创建 src/hello.ts，导出 greet 函数
[AI]: 已创建，包含 greet 函数返回 Hello World

---
当前任务：加个 farewell 函数返回 Goodbye

这样 Agent 知道前面创建过什么文件、做了什么修改，不需要用户重复描述。

为什么历史拼成文本而不是多轮 messages？ OpenAI 格式要求工具调用后使用 role: "tool" 消息携带 tool_call_id。前端存储的历史消息只有纯文本 content，没有 tool_calls 结构。拼成文本是安全的折中方案。

export const readFileTool: Tool = {
  name: "read_file",
  description: "读取文件内容，支持行号范围",
  schema: {
    type: "object",
    properties: {
      path:   { type: "string", description: "文件路径" },
      offset: { type: "number", description: "起始行号，默认 1" },
      limit:  { type: "number", description: "读取行数，默认 200" },
    },
    required: ["path"],
  },
  async run(args) {
    const fp = String(args.path);
    if (!fs.existsSync(fp)) return `错误：文件不存在 —— ${fp}`;
    const lines = fs.readFileSync(fp, "utf-8").split("\n");
    const offset = Number(args.offset ?? 1);
    const limit  = Number(args.limit  ?? 200);
    const end = Math.min(offset + limit, lines.length);
    const result = lines.slice(offset - 1, end);
    // 返回：[path] L1-L50 / 200 行 + 带行号的代码
    return [`[${fp}] L${offset}-L${end} / ${lines.length} 行`,
            ...result.map((l, i) => `${offset + i}: ${l}`)].join("\n");
  },
};

九、工具：write_file

参数类型说明
pathstring文件路径（相对路径自动 resolve）
contentstring完整内容
export const writeFileTool: Tool = {
  name: "write_file",
  description: "写入文件（覆盖），自动创建父目录",
  schema: {
    type: "object",
    properties: {
      path:    { type: "string", description: "文件路径" },
      content: { type: "string", description: "完整内容" },
    },
    required: ["path", "content"],
  },
  async run(args) {
    const fp = path.resolve(String(args.path));  // → 绝对路径
    const content = String(args.content);
    fs.mkdirSync(path.dirname(fp), { recursive: true });
    fs.writeFileSync(fp, content, "utf-8");
    const lines = content.split("\n");
    const preview = lines.slice(0, 15).join("\n");  // 前 15 行预览
    const more = lines.length > 15 ? `\n...（共 ${lines.length} 行）` : "";
    return `✅ 文件已创建: ${fp}\n${lines.length} 行，${content.length} 字节\n\n--- 内容预览 ---\n${preview}${more}`;
  },
};

设计要点：path.resolve() 保证返回绝对路径；前 15 行预览嵌入返回结果，用户不用离开聊天就能看到内容。

十、工具：edit_file 核心亮点

参数类型说明
pathstring文件路径
old_stringstring要被替换的文本，必须在文件中唯一
new_stringstring替换后的文本

为什么需要 edit_file：write_file 是全量覆盖，编辑一行要重写整个文件——容易丢内容。edit_file 只替换匹配的片段。

匹配策略（四层）

edit_file(path, old_string, new_string) │ ├─ 读取文件内容 │ ├─ 精确匹配 old_string │ ├─ 出现 1 次 → 替换 ✓ │ └─ 出现 >1 次 → 报错：不唯一，需增加上下文 │ ├─ 精确匹配 失败？ │ └─ trim() 后匹配 │ ├─ 出现 1 次 → 替换（标注"宽松匹配"）✓ │ └─ 出现 0 次 → 报错 + 返回文件前 500 字符供调试 
export const editFileTool: Tool = {
  name: "edit_file",
  description: "精准编辑文件——查找并替换指定文本",
  schema: {
    type: "object",
    properties: {
      path:       { type: "string", description: "文件路径" },
      old_string: { type: "string", description: "要替换的文本（需唯一）" },
      new_string: { type: "string", description: "新文本" },
    },
    required: ["path", "old_string", "new_string"],
  },
  async run(args) {
    const fp = path.resolve(String(args.path));
    const oldStr = String(args.old_string);
    const newStr = String(args.new_string);
    if (!fs.existsSync(fp)) return `错误：文件不存在 —— ${fp}`;

    const content = fs.readFileSync(fp, "utf-8");

    // 统计出现次数
    let count = 0, pos = 0;
    while ((pos = content.indexOf(oldStr, pos)) !== -1) { count++; pos++; }

    if (count === 0) {
      // 宽松匹配：trim 后重试
      const trimmed = oldStr.trim();
      let tCount = 0, tPos = 0;
      while ((tPos = content.indexOf(trimmed, tPos)) !== -1) { tCount++; tPos++; }
      if (tCount === 1) {
        fs.writeFileSync(fp, content.replace(trimmed, newStr), "utf-8");
        return `✅ 已编辑: ${fp}（宽松匹配）`;
      }
      return `错误：未找到匹配。\n文件前 500 字符：\n${content.slice(0, 500)}`;
    }

    if (count > 1) return `错误：出现了 ${count} 次，不唯一。请增加上下文行。`;

    fs.writeFileSync(fp, content.replace(oldStr, newStr), "utf-8");
    return `✅ 已编辑: ${fp}\n替换了 1 处匹配`;
  },
};

参数类型说明

递归遍历目录，跳过 . 开头目录和 node_modules，每个文件逐行匹配。最多 50 个结果。不支持正则时自动转义为纯文本搜索。

function safeRegex(p: string): RegExp {
  try { return new RegExp(p, "gi"); }
  catch {
    // 不是合法正则 → 转义特殊字符后当纯文本搜
    return new RegExp(p.replace(/[.*+?^${}()|[\]\\]/g, "\\$&"), "gi");
  }
}

十五、完整数据流

一次「创建 hello.ts」从用户输入到页面渲染的完整链路：

① 用户输入 "创建 src/hello.ts，导出 greet 函数" │ ② page.tsx: POST /api/chat { messages, model, baseUrl } │ ③ route.ts: task="创建...", history=[], new PiAgent() │ ④ core.ts: run() → 构建 system prompt + task │ ⑤ callLLM() → POST /chat/completions │ body: { model, messages, tools: [...5个], tool_choice: "auto" } │ ⑥ LLM 返回 tool_calls[{ function: { name:"write_file", arguments:'{"path":"src/hello.ts","content":"export function greet()..."}' }}] │ ⑦ Agent: tools.get("write_file") → JSON.parse(arguments) → emit("action") → route 转 SSE → 前端显示工具调用卡片 │ ⑧ writeFileTool.run() → fs.writeFileSync() → 文件落盘 → 返回 "✅ 文件已创建: /absolute/path\n3行,61字节\n---内容预览---" │ ⑨ emit("observation") → route 转 SSE → 前端显示文件创建结果 │ ⑩ 结果以 role:"tool" 消息推入 messages，继续下一轮 │ ⑪ LLM: 没有 tool_calls → content="已创建 greet 函数" │ ⑫ emit("answer") → route 追加文件汇总 → SSE [DONE] │ ⑬ 前端: ReactMarkdown 渲染完整回答

十六、关键设计决策

为什么用 function calling 而不是文本解析？

v1 用文本 ReAct（Thought:/Action:/Action Input: 正则匹配），DeepSeek 和 Gemma 不遵守格式，直接把代码写在回答里。function calling 是 LLM 原生支持的机制，tool_calls 是结构化的，零格式错误。

为什么 temperature=0？

工具调用要求确定性。temperature>0 可能导致同一问题两次调用不同工具，不可调试。

为什么 history 拼成文本而不是多轮 messages？

OpenAI 格式要求 role: "tool" 消息携带 tool_call_id。前端存储的历史消息只有纯文本 content，没有 tool_calls 结构。拼成文本是最安全的折中。

为什么 edit_file 用 old_string 替换而不是 diff/patch？

diff 格式对人类友好但对 LLM 复杂——需要精确计算行号偏移。old_string → new_string 查找替换对 LLM 更自然：它已经看过文件内容，知道要改哪一段。
